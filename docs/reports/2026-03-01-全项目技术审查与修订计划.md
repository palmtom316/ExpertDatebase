## 一、总体评价

项目自 2026-02-28 上次审计（36 tests）以来已大幅推进，所有核心功能链路均已实现。架构设计清晰（Repository Pattern、Feature Flag、Circuit Breaker、BYOK），测试覆盖也较为系统。

**然而，当前状态与"正式上线"之间仍存在 12 项明确差距**，分布在安全、运维、可观测性、检索质量、前端工程性等维度。

## 二、差距清单（按严重程度）

### 🔴 CRITICAL（阻塞上线）

| # | 问题 | 位置 | 说明 |
|---|------|------|------|
| C-1 | **Auth 默认关闭** | `docker-compose.yml:24`，`.env.example` | `AUTH_ENABLED=false` 默认值，部署时若遗漏设置，所有 admin 接口直接暴露，返回 `system_admin` 权限 |
| C-2 | **默认明文密码写死在 compose** | `docker-compose.yml:14,12,25` | `postgres:postgres`、`minio123` 明文在版本库，`secrets_guard.py` 仅在 `APP_ENV=production` 时校验，部署时极易漏配 |
| C-3 | **Upload 全量读入内存** | `upload.py:259` `content = await file.read()` | 50MB 最大文件全量读入内存，高并发下内存溢出风险；无流式处理 |
| C-4 | **MinerU 真实集成缺失** | `worker/mineru_client.py` | compose 中无 MinerU 服务；stub 仅返回空结构，真实扫描 PDF 处理能力为零 |

### 🟠 HIGH（严重影响功能可用性）

| # | 问题 | 位置 | 说明 |
|---|------|------|------|
| H-1 | **Chat/Search 实体索引永远为空** | `chat.py:19-25`，`search.py`（同模式） | `_EmptyEntityIndex` 硬编码，`filter_parser.py` 的人名/角色过滤链路永不触发，关系过滤能力完全失效 |
| H-2 | **Keyword 检索是全量 scroll + 内存过滤** | `search_service.py:212-236` | `keyword_search` 先 scroll 最多 3000 条再内存 TF 打分，数据量增大后性能灾难，不具备生产规模 |
| H-3 | **LLM 路由 tier 解析逻辑有缺陷** | `llm_router.py:60-89` | `LLM_TIER_PROVIDER_{tier_name}` 的 env key 拼写依赖 tier_name 值，compose 中没有对应环境变量，实际 tier failover 链路不生效，所有请求都降级 stub |
| H-4 | **无 CI/CD 流水线** | 项目根目录 | 零自动化测试/构建/部署；无法保证主干代码质量门禁 |
| H-5 | **数据库迁移无自动化** | `alembic/versions/` | `alembic upgrade head` 必须手动运行，容器启动时不自动迁移，运维操作遗漏风险高 |
| H-6 | **无限速/防滥用保护** | 所有路由 | 无 rate limiting，上传、chat、search 接口可被无限打满 |

### 🟡 MEDIUM（影响系统质量与运维）

| # | 问题 | 位置 | 说明 |
|---|------|------|------|
| M-1 | **无可观测性基础设施** | 全服务 | 仅有 `print()` 语句，无结构化日志（loguru/structlog），无 Prometheus metrics，无分布式追踪；故障排查几乎靠肉眼看容器日志 |
| M-2 | **前端单文件 1500+ 行** | `src/frontend/src/App.vue` | 违反 800 行上限规范，无组件拆分，维护风险极高；`dist/` 目录提交到版本库（不推荐）|
| M-3 | **Embedding 失败无声降级** | `search_service.py:332-334` | `embed_text` 出现任何异常静默降级到 stub hash，查询向量维度可能与索引不匹配，且没有任何日志告警 |
| M-4 | **P1/P2 RAG 能力未激活，检索质量指标未验证** | `.env.example` | PG-BM25、Sirchmunk、GraphRAG、LangExtract 均 disabled；未验证 Hit@10≥0.75、MRR≥0.45 指标 |
| M-5 | **pyproject.toml 依赖声明不完整** | `pyproject.toml` | `uvicorn`、`pytest`、`httpx`、`alembic` 等核心依赖缺失，新环境装包不稳定 |
| M-6 | **Worker healthcheck 是假检查** | `docker-compose.yml:76,92` | worker/scheduler healthcheck 只跑 `python -c "print('ok')"` ，不实际检查服务健康 |
| M-7 | **CORS 允许来源过于宽松** | `main.py:20` | 默认允许 `localhost:5500` + 任意注入，生产环境应强制配置精确域名 |
| M-8 | **`page_type` 判断逻辑仅取第一个匹配** | `runner.py:199-204` | 整个文档只用一个 `page_type` 标注所有 chunk，应当 per-chunk 或 per-page 标注 |

### 🔵 LOW（代码质量与工程规范）

| # | 问题 | 位置 | 说明 |
|---|------|------|------|
| L-1 | **Secret 泄露风险**：`llm_api_key` 等可能经由 log 路径泄漏 | `llm_router.py:277` | `metadata_json` 含 `attempted_providers` 等，需确认 runtime_config 中 API key 不进入持久化日志 |
| L-2 | **无压测/性能基准** | — | 无任何吞吐量测量，不知道并发上限 |
| L-3 | **Qdrant 集合索引未配置 payload index** | `worker/qdrant_index.py` | `doc_id`、`version_id` 等高频过滤字段若无 payload index，filter 走全量扫描 |
| L-4 | **`assert` 用于生产关键路径** | `llm_router.py:245` | `assert semaphore is not None` 在 python -O 模式下被优化掉，应改为显式检查 |
| L-5 | **前端 API base 硬编码风险** | `App.vue` | 前端 API_BASE 若 hardcoded 为 localhost，容器化部署时需手动修改 |

## 三、修订计划（分阶段实施）

### Phase 0：上线阻塞修复（预计 3-4 天）

> 目标：消除 C-1~C-4 和 H-1~H-6，达到可以安全部署的最低要求

| 任务编号 | 任务描述 | 涉及文件 |
|---------|---------|---------|
| P0-1 | **Auth 默认开启**：将 compose 中 `AUTH_ENABLED` 默认值改为 `true`，并在 `.env.example` 强制要求配置 `AUTH_TOKENS_JSON` | `docker-compose.yml`, `.env.example` |
| P0-2 | **Secrets 外置**：将 compose 中的 postgres/minio 密码改为 env var 引用（`${POSTGRES_PASSWORD}`），更新 `secrets_guard.py` 在非生产环境也输出警告 | `docker-compose.yml`, `secrets_guard.py` |
| P0-3 | **上传改流式处理**：`upload.py` 改用 `SpooledTemporaryFile` 或 `UploadFile.chunks()` 流式读取，校验在 chunk 层进行；增加对文件大小的 header pre-check | `upload.py` |
| P0-4 | **数据库自动迁移**：api-server Dockerfile 或 compose entrypoint 增加 `alembic upgrade head` 启动前调用 | `services/api-server/Dockerfile` 或 compose entrypoint |
| P0-5 | **实体索引接入**：将 `chat.py` 和 `search.py` 的 `_EmptyEntityIndex` 替换为真实的 `EntityIndex`（读取 PG entity_dictionary 表），使人名/角色过滤链路生效 | `chat.py`, `search.py`, 新增 `entity_index.py`（api-server 侧） |
| P0-6 | **LLM tier 路由修复**：修复 `_providers_for_task` 中 env key 拼接逻辑，确保 `LLM_PROVIDER=openai` 时正确读取 BYOK key；在 compose 中补全 tier 环境变量 | `llm_router.py`, `docker-compose.yml` |
| P0-7 | **基础 Rate Limiting**：使用 `slowapi` 或 middleware 对 `/api/upload`（5次/分）、`/api/chat`（30次/分）、`/api/search`（60次/分）加限速 | `main.py`，新增 middleware |
| P0-8 | **CI/CD 基础流水线**：新增 `.github/workflows/ci.yml`，在 push/PR 时运行 `pytest -q`，blocking merge | `.github/workflows/ci.yml` |

### Phase 1：核心质量补齐（预计 4-5 天）

> 目标：消除 M-1~M-8，达到生产质量标准

| 任务编号 | 任务描述 | 涉及文件 |
|---------|---------|---------|
| P1-1 | **结构化日志**：引入 `structlog` 或 `loguru`，替换所有 `print()`，统一输出 JSON 格式；关键事件（任务入队/完成/失败、LLM调用、检索路由）必须有日志 | 全服务 |
| P1-2 | **Keyword 检索性能**：将 Qdrant keyword_search 改为真正的 Qdrant fulltext 索引或启用 PG-BM25（已有实现），彻底消除全量 scroll + 内存过滤 | `search_service.py`, `.env.example` |
| P1-3 | **Qdrant Payload Index**：在 `qdrant_index.py` 中为 `doc_id`、`version_id`、`doc_type`、`val_voltage_kv` 等高频字段创建 payload index | `worker/qdrant_index.py` |
| P1-4 | **pyproject.toml 补全**：将 `uvicorn`, `alembic`, `pytest`, `httpx`, `structlog`, `slowapi` 等加入依赖声明 | `pyproject.toml` |
| P1-5 | **前端组件拆分**：将 `App.vue` 拆分为至少 6 个子组件（UploadPanel, DocList, ChatPanel, SearchPanel, AdminPanel, SettingsDrawer），每个 < 300 行 | `src/frontend/src/` |
| P1-6 | **Worker/Scheduler 真实 healthcheck**：改为实际检查 Redis 连接和最近处理时间 | `docker-compose.yml` |
| P1-7 | **per-chunk page_type 标注**：修复 runner.py 的 page_type 判断逻辑，改为 per-chunk 基于 `chapter_id` 或 `page_start` 做精细标注 | `worker/runner.py` |
| P1-8 | **Embedding 降级告警**：embed_text 失败时记录 WARNING 日志，不再静默降级 | `search_service.py` |

### Phase 2：检索质量达标（预计 3-5 天）

> 目标：验证 Hit@10≥0.75、MRR≥0.45、EvidenceHitRate@10≥0.80

| 任务编号 | 任务描述 | 涉及文件 |
|---------|---------|---------|
| P2-1 | **激活 PG-BM25 稀疏检索**：设置 `ENABLE_PG_BM25=1`，验证 pg_bm25.py 实现的 tsvector 索引；补充 chunks 表的 `to_tsvector` 触发器或物化列 | `pg_bm25.py`, migration |
| P2-2 | **Structured Lookup 激活**：设置 `ENABLE_STRUCTURED_LOOKUP=1`，验证证书号/标准号/人员资质查询精确命中路径 | `structured_lookup.py` |
| P2-3 | **构建评测基准集**：整理至少 30 条真实问题集到 `datasets/v1.0/`，运行 `eval_retrieval.py`，输出基线报告 | `datasets/`, `eval_retrieval.py` |
| P2-4 | **MinerU 集成验证**：配置真实 MinerU API，端到端测试真实 PDF 处理，验证 mineru_json_key/md_key 正确写入 MinIO | `worker/mineru_client.py` |

### Phase 3：上线收口（预计 2-3 天）

> 目标：完成上线前检查清单，执行压测和故障演练

| 任务编号 | 任务描述 |
