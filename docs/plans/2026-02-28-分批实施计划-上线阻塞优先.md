# ExpertDatebase 分批实施计划（上线阻塞优先）

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 将 Phase 1~5 转为可执行批次，先完成上线阻塞项，再推进功能完善与上线准备。  
**Architecture:** 采用“阻塞项先行”的分层推进：`B0 阻塞` -> `B1 核心功能` -> `B2 前端与评测闭环` -> `B3 运维上线收口`。  
**Tech Stack:** FastAPI, SQLAlchemy/Alembic, PostgreSQL, Redis, Qdrant, MinIO, Worker Pipeline, Vanilla JS Frontend。

---

## 批次划分

### B0（上线阻塞，必须先完成）

#### B0.1 安全与入口防护
- 范围：
  - RBAC 鉴权（系统管理员/投标工程师/公司员工）
  - `/api/admin/*` 强制管理员权限
  - 上传文件类型/大小限制（PDF only + size limit）
  - 基础 secrets 管理约束（禁止默认凭据启动生产）
- 验收标准：
  - 未带认证访问 `/api/admin/jobs/*` 返回 `401/403`
  - 非管理员访问 admin 路径返回 `403`
  - 超限或非 PDF 上传被拒绝（`413/415`）
  - 新增自动化测试覆盖以上行为

#### B0.2 Worker 真实处理链路
- 范围：
  - `quality_gate`（A/B/C）
  - `classify` 文档分类
  - 章节退化策略 C 完善（按页切 -> 合并）
  - `table_struct` 三类关键表格结构化 + LLM 修复兜底
  - `ie_extract` 补齐 qualification/person/equipment/standard
  - `entity_index` 持久化到 Postgres（替代 hash）
  - 真实 Embedding API 客户端替换 stub
- 验收标准：
  - 给定真实样本 PDF，流水线产出完整中间态（blocks/tables/chapters/chunks/assets）
  - 三类关键表格至少各有 1 个样例可结构化落库
  - entity id 可跨进程稳定复现
  - embedding 调用可切换真实 provider，失败时有明确降级策略与日志

#### B0.3 API 可靠性与调度
- 范围：
  - `chat_orchestrator` 真实 evidence 拼接（before/excerpt/after）
  - LLM Tier 级联降级 + circuit breaker
  - 脱敏机制（外发默认脱敏）
  - LLM 并发控制（3）
  - scheduler 服务（巡检/评测调度）
- 验收标准：
  - QA 返回引用与可展开证据，证据来自真实命中文本上下文
  - provider 故障时按 tier 自动切换并记录
  - 熔断阈值触发后进入冷却期，冷却后自动恢复
  - 并发压测下 LLM 并发数不超过 3
  - scheduler 可周期触发 expiry_scan 和 eval 调度任务

### B1（核心产品闭环）

#### B1.1 前端核心界面
- 范围：Chat UI、Review UI、文档列表 API 化、一键加入评测集
- 验收标准：
  - 文档列表来自后端 API，不再依赖静态 mock
  - Review 支持 PDF 跳页、block/table 证据定位与摘录选择
  - 一键加入评测集可写入 manifest/input/truth 基础文件

#### B1.2 管理端评测仪表盘
- 范围：4 子页（IE/Table/QA/Retrieval）+ run/detail API
- 验收标准：
  - 可查看 run 列表、结果明细、分项指标与 diff
  - 关键趋势（失败率/延迟/质量分）可视化

### B2（评测体系完善）

#### B2.1 数据集与评测执行
- 范围：`datasets/v1.0` 目录规范落地；`eval_runner` 复用 LLMRouter 执行
- 验收标准：
  - manifest 驱动执行 `model x sample`
  - 产物写入 `outputs/{run_id}` + MinIO + `eval_run/eval_result`

#### B2.2 评分与 diff 完整化
- 范围：容差规则、四类 diff_report 完整结构、路由联动建议
- 验收标准：
  - 金额容差 ±1%、kV 精确命中、线路 ±0.1km 生效
  - IE/TABLE/QA/RET 四类 diff 报告字段齐全
  - 评分下降触发模型切换建议事件

### B3（运维与上线收口）

#### B3.1 Docker/配置/迁移
- 范围：compose 加 scheduler、secrets 管理、migration 补全
- 验收标准：
  - `docker compose up` 可一键拉起全链路（含 scheduler）
  - 无默认明文凭据进入生产配置
  - 所有新增表均通过 Alembic 管理

#### B3.2 观测与发布准备
- 范围：健康检查、告警、日志级别、资源限制、备份策略
- 验收标准：
  - 每个核心服务有健康检查端点/探针
  - 关键错误有告警通道
  - 提供上线 runbook 与回滚方案
